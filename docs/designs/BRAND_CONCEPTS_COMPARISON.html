Visual identity review — Ride Optimizer

Three ways to make this look worth paying for

The app has real engineering behind it: Fréchet-distance route matching, live weather scoring, skeleton loading states, WCAG-conscious focus rings. None of that shows on screen. Visually it reads as an unstyled Bootstrap admin panel with one gradient standing in for a brand. Below is an honest read of what's there now, then three complete identities built from the app's own material — instrumentation, terrain, and weather — each carried through the navigation, the ride decision card, and a route row.

Where it stands today

One gradient, doing every job

#667eea → #764ba2 covers the navbar, the hero banner, badges, card borders, and focus rings. When a color means "brand," "focus," "selected," and "this is the optimal route" all at once, it stops reading as a decision and starts reading as a default nobody removed.

No point of view on numbers

Temperature, match score, distance, elevation gain — the things a cyclist actually scans for — are set in the same system-font weight as body copy. On a data-driven app, the numbers should look designed, not left over.

Bootstrap Icons, Bootstrap radii, Bootstrap shadows

The 8px card radius, the bi-* icon set, the default drop shadows — these are the framework's defaults, unedited. Nothing about the surface says "cycling" or "weather" specifically; it could be an invoicing tool.

Good bones, generic skin

Touch-target sizing, :focus-visible handling, skeleton loaders, reduced-motion support — genuinely well built and worth keeping. Each concept below reuses that structure and replaces only the surface.
